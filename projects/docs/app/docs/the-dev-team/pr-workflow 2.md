# PR Workflow

When a task passes every validation gate, the orchestrator opens a pull request on behalf of the bot account. The PR is the hand-off point between the autonomous system and a human reviewer — and everything that happens after merge (CI, deploy) is outside THE Dev Team.

This page describes how tasks become PRs, what goes in the description, and how review comments are handled.

## Task → commit → push → PR

Phase 6 of the [Execution Loop](execution-loop.md#phase-6--submit):

```
git add -A
git commit -m "feat(users): add profile page

Closes #123
Task: abc123
Cost: $0.87
"
git push origin the-dev-team/feature/abc123
gh pr create --title "[THE Dev Team] Add user profile page" \
             --body "$(build-pr-body abc123)" \
             --base main \
             --head the-dev-team/feature/abc123
```

The commit message follows conventional commits. It includes the task id, the source issue (if any), and the final cost. The implementation aims for **one commit per task** — the bugfixer amends rather than stacking commits in most cases (exceptions: when the rework touches files the original implementer didn't modify).

## PR description structure

The `PrManagerService.buildPRBody(task)` method assembles a structured description from the task state directory. The template is stable so humans can skim PRs quickly and the dashboard can parse them:

```markdown
## Summary
{task description — the original prompt}

## Changes
{git diff --stat main...HEAD}

## Validation Results

### Tests
- Unit tests: passed (backend: 142, frontend: 87)
- Integration tests: passed (12 specs)
- E2E tests: passed (5 specs)

### Accessibility
WCAG AA: 0 violations across 3 pages

### Performance
| Endpoint | Baseline p95 | Current p95 | Delta |
|----------|-------------|-------------|-------|
| GET /api/users | 65ms | 62ms | -5% |
| GET /api/users/:id | 42ms | 44ms | +5% |

### Design Review
No blocking issues. 1 advisory: avatar shadow is slightly darker than design system standard.

## Screenshots
### Mobile (375x812)
![mobile](.the-dev-team/state/abc123/screenshots/mobile-profile.png)

### Tablet (768x1024)
![tablet](.the-dev-team/state/abc123/screenshots/tablet-profile.png)

### Desktop (1440x900)
![desktop](.the-dev-team/state/abc123/screenshots/desktop-profile.png)

## Environment
- Branch: `the-dev-team/feature/abc123`
- Task ID: `abc123`
- Total Cost: $0.87
- Sandbox: `env-abc123` (kept alive for review)

---
*Generated by THE Dev Team*
```

Every section is populated from files in `.the-dev-team/state/{task-id}/`:

- **Tests**, **Accessibility**, **Performance**, **Design Review** — from `gate-results/*.json`
- **Screenshots** — from `screenshots/after/`
- **Changes** — live `git diff --stat`
- **Cost** — accumulated on the task object throughout the run

If a section has no data (e.g. `touchesFrontend: false` means no screenshots), it is omitted rather than left empty.

## Screenshot hosting

Screenshots are committed to the PR branch under `.the-dev-team/state/{task-id}/screenshots/` so they render directly in the PR description using relative links. This keeps the evidence with the branch and means anyone checking out the branch has the screenshots too. When the PR is merged, the squash commit drops the screenshots from `main` (or they can be selectively excluded via a `.gitignore` rule).

## Review comment detection

After the PR is opened, the orchestrator's scheduler polls GitHub every 2 minutes for new review comments on open PRs authored by the bot:

```typescript
@Cron('*/2 * * * *')
async checkForReviewComments() {
  const prs = await this.getOpenBotPRs();
  for (const pr of prs) {
    const newComments = await this.getUnprocessedComments(pr.number);
    if (newComments.length > 0) {
      await this.createReworkTask(pr, newComments);
    }
  }
}
```

"Unprocessed" is tracked by persisting the last-seen comment id per PR in `.the-dev-team/state/{task-id}/processed-comments.json`. The scheduler only creates a new rework task when there are comments newer than the last one processed.

Polling is used instead of webhooks because it's simpler and the 2-minute latency is acceptable. Webhooks can be added later if the volume ever warrants it.

## The rework loop

A rework task is a normal task with `source: 'pr_feedback'` and the original `branch` preserved. The execution loop treats it slightly differently:

1. **Setup** — `git worktree add` uses the **existing** PR branch instead of creating a new one
2. **Implement** — only the implementer runs (not the architect), with the review comments as the prompt
3. **Build + Deploy** — normal gate sequence; the sandbox is either recreated or reused
4. **Test** — normal gate sequence
5. **Review + Fix** — the reviewer re-runs against the updated diff
6. **Submit** — `git push` (no force), then `gh pr comment` with a summary of what changed
7. **Cleanup** — same as a normal task

Key differences from a fresh task:

- Same branch, same PR number
- Only the implementer sees the review comments (the reviewer re-reviews the whole diff, not just the comments)
- The PR description is updated via `gh pr edit` with refreshed validation results
- A new comment is added summarising what was changed

```
New commits pushed. Rework summary:
- Fixed error handling in UserService.create (reviewer comment #1)
- Extracted CreateUserDto (reviewer comment #2, advisory)

Validation results refreshed:
- Unit tests: passed
- Integration tests: passed
- E2E tests: passed
- Performance: within thresholds
```

## Max rework iterations

To avoid indefinite back-and-forth, there's a per-PR rework cap (default 3). After 3 rework cycles the task is escalated: the orchestrator stops polling for new comments and emits a `task:escalated` event. A human then decides whether to continue manually, close the PR, or tweak the prompt.

## Approval detection

A separate scheduled job checks for PR approvals every 5 minutes:

```typescript
@Cron('*/5 * * * *')
async checkForApprovals() {
  const prs = await this.getOpenBotPRs();
  for (const pr of prs) {
    if (await this.isApproved(pr)) {
      // Stop the rework poller for this PR
      // Mark the task as awaiting_merge
    }
  }
}
```

On detecting approval the orchestrator doesn't merge — it just updates state. When the PR is actually merged (by a human), the monitor role picks up CI results and the sandbox is cleaned up by the stale-environment reaper if it hasn't been already.

## Agent never merges

For completeness: there is no code path in the orchestrator that calls `gh pr merge`. The bot account also lacks permission to merge to protected branches. Both are required for the safety model — see [Safety Model](safety-model.md#pr-merge-is-human-only).

## Monitor role after merge

Post-merge, the monitor role takes over. On a schedule it:

1. Lists recent merges to `main`
2. Checks the CI status for each
3. Tails deployment logs for any regression
4. Files a new task with `source: 'ci_failure'` if something breaks, which flows back through the normal execution loop as a fix

This closes the loop — a regression caused by the team's own PR is picked up, triaged, and fixed without human intervention (subject to the same safety and review constraints).
