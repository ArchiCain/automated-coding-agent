# Skill: Database

You are working with database schema changes, entity definitions, migrations, and
data management using **TypeORM** with **PostgreSQL**.

---

## Entity Definition

Entities live in their feature directory:

```
src/features/{feature}/entities/{entity-name}.entity.ts
```

### Entity Pattern

```typescript
import {
  Entity,
  PrimaryGeneratedColumn,
  Column,
  CreateDateColumn,
  UpdateDateColumn,
  Index,
  ManyToOne,
  JoinColumn,
} from 'typeorm';

@Entity('feature_items')
export class FeatureItem {
  @PrimaryGeneratedColumn('uuid')
  id: string;

  @Column({ type: 'varchar', length: 255 })
  @Index()
  name: string;

  @Column({ type: 'text', nullable: true })
  description: string | null;

  @Column({ type: 'varchar', length: 50, default: 'active' })
  status: string;

  @Column({ type: 'jsonb', nullable: true })
  metadata: Record<string, unknown> | null;

  @Column({ type: 'integer', default: 0 })
  sortOrder: number;

  @ManyToOne(() => ParentEntity, { onDelete: 'CASCADE' })
  @JoinColumn({ name: 'parent_id' })
  parent: ParentEntity;

  @Column({ type: 'uuid' })
  parentId: string;

  @CreateDateColumn()
  createdAt: Date;

  @UpdateDateColumn()
  updatedAt: Date;
}
```

### Entity Rules

1. **Primary keys** — Always UUID, generated by the database.
2. **Timestamps** — Every entity gets `createdAt` and `updatedAt`.
3. **Nullable** — Columns are NOT NULL by default. Explicitly mark nullable columns.
4. **Column types** — Always specify the SQL type explicitly. Do not rely on TypeORM inference.
5. **Indexes** — Add indexes on columns used in WHERE clauses and foreign keys.
6. **Relations** — Define both the decorator (`@ManyToOne`) and the foreign key column (`parentId`).
7. **Table names** — Use `snake_case`, plural (e.g., `feature_items`, `user_profiles`).
8. **Column names** — TypeORM auto-converts camelCase to snake_case. Let it.

## Migration Creation

### Generating a Migration

After defining or modifying an entity, generate a migration:

```bash
# Generate migration from entity changes
npx typeorm migration:generate src/migrations/{MigrationName} -d src/data-source.ts

# If the data source path differs, check package.json for the correct command
npm run migration:generate -- -n {MigrationName}
```

### Manual Migration

When the generator does not produce the right output (e.g., data migrations, custom indexes):

```bash
npx typeorm migration:create src/migrations/{MigrationName}
```

### Migration Pattern

```typescript
import { MigrationInterface, QueryRunner } from 'typeorm';

export class AddUserProfileTable1700000000000 implements MigrationInterface {
  name = 'AddUserProfileTable1700000000000';

  public async up(queryRunner: QueryRunner): Promise<void> {
    await queryRunner.query(`
      CREATE TABLE "user_profiles" (
        "id" uuid NOT NULL DEFAULT uuid_generate_v4(),
        "user_id" uuid NOT NULL,
        "display_name" varchar(255) NOT NULL,
        "bio" text,
        "avatar_url" varchar(500),
        "created_at" TIMESTAMP NOT NULL DEFAULT now(),
        "updated_at" TIMESTAMP NOT NULL DEFAULT now(),
        CONSTRAINT "PK_user_profiles" PRIMARY KEY ("id"),
        CONSTRAINT "UQ_user_profiles_user_id" UNIQUE ("user_id"),
        CONSTRAINT "FK_user_profiles_user_id" FOREIGN KEY ("user_id")
          REFERENCES "users"("id") ON DELETE CASCADE
      )
    `);

    await queryRunner.query(`
      CREATE INDEX "IDX_user_profiles_user_id" ON "user_profiles" ("user_id")
    `);
  }

  public async down(queryRunner: QueryRunner): Promise<void> {
    await queryRunner.query(`DROP INDEX "IDX_user_profiles_user_id"`);
    await queryRunner.query(`DROP TABLE "user_profiles"`);
  }
}
```

### Migration Rules

1. **Always reversible** — Every `up` must have a corresponding `down`.
2. **Idempotent** — Use `IF NOT EXISTS` / `IF EXISTS` where possible.
3. **Backward compatible** — Migrations should not break the currently deployed version.
4. **No data loss** — Never drop columns with data without a data migration step first.
5. **Order matters** — Migrations run in timestamp order. Name them accordingly.
6. **One concern per migration** — Do not mix table creation with data seeding.

### Backward Compatibility Strategy

For breaking schema changes, use a two-phase approach:

**Phase 1 (additive):** Add the new column/table, keep the old one.
```sql
ALTER TABLE "users" ADD COLUMN "email_verified" boolean DEFAULT false;
```

**Phase 2 (cleanup):** After all services are updated, remove the old column.
```sql
ALTER TABLE "users" DROP COLUMN "old_email_status";
```

## Validating Schema State

After running migrations, verify the schema:

```bash
# Check which migrations have been applied
task env:db:query TASK_ID=$TASK_ID QUERY="SELECT * FROM migrations ORDER BY id DESC LIMIT 10;"

# Verify table structure
task env:db:query TASK_ID=$TASK_ID QUERY="
  SELECT column_name, data_type, is_nullable, column_default
  FROM information_schema.columns
  WHERE table_name = 'user_profiles'
  ORDER BY ordinal_position;
"

# Verify indexes
task env:db:query TASK_ID=$TASK_ID QUERY="
  SELECT indexname, indexdef
  FROM pg_indexes
  WHERE tablename = 'user_profiles';
"

# Verify foreign keys
task env:db:query TASK_ID=$TASK_ID QUERY="
  SELECT
    tc.constraint_name,
    tc.table_name,
    kcu.column_name,
    ccu.table_name AS foreign_table_name,
    ccu.column_name AS foreign_column_name
  FROM information_schema.table_constraints AS tc
  JOIN information_schema.key_column_usage AS kcu ON tc.constraint_name = kcu.constraint_name
  JOIN information_schema.constraint_column_usage AS ccu ON ccu.constraint_name = tc.constraint_name
  WHERE tc.constraint_type = 'FOREIGN KEY' AND tc.table_name = 'user_profiles';
"
```

## Seed Data

Seed data is for development and testing environments only.

### Seed File Location

```
src/seeds/
  {entity-name}.seed.ts
```

### Seed Pattern

```typescript
import { DataSource } from 'typeorm';
import { FeatureItem } from '../features/feature/entities/feature-item.entity';

export async function seedFeatureItems(dataSource: DataSource): Promise<void> {
  const repository = dataSource.getRepository(FeatureItem);

  const existingCount = await repository.count();
  if (existingCount > 0) {
    return; // Already seeded, skip
  }

  const items = repository.create([
    { name: 'Item A', description: 'First seed item', status: 'active' },
    { name: 'Item B', description: 'Second seed item', status: 'active' },
    { name: 'Item C', description: 'Third seed item', status: 'draft' },
  ]);

  await repository.save(items);
}
```

### Seed Rules

1. **Idempotent** — Seeds check if data exists before inserting.
2. **Realistic** — Seed data should represent realistic usage, not `test123`.
3. **Minimal** — Enough data to verify the UI works, not thousands of rows.
4. **No production data** — Never include real user data, emails, or PII.
5. **Stable IDs** — Use deterministic UUIDs for seed data if other seeds reference them.

## Common Patterns

### Adding a Column to an Existing Table

1. Update the entity file with the new `@Column` decorator.
2. Generate a migration: `npx typeorm migration:generate`.
3. Review the generated SQL. Ensure it is additive (no DROP).
4. Run: `task env:db:migrate TASK_ID=$TASK_ID`.
5. Verify: query `information_schema.columns`.

### Creating a New Feature Table

1. Create the entity file in `src/features/{name}/entities/`.
2. Register the entity in the feature module: `TypeOrmModule.forFeature([Entity])`.
3. Generate a migration.
4. Review the generated SQL for correctness.
5. Run the migration and verify.

### Renaming a Column

1. **Phase 1:** Add new column, copy data, keep old column.
2. Deploy. Verify both columns work.
3. **Phase 2:** Update code to use new column only.
4. **Phase 3:** Drop old column in a separate migration.
